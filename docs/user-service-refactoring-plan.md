# 高考志愿填报系统 - 用户服务重构方案

## 1. 重构背景

根据《全栈代码库综合审计报告》的分析，用户服务 (`user-service`) 存在以下严重问题：

1. **架构缺陷**：通过CGO深度耦合了设备指纹和许可证管理的C++模块，导致：
   - 平台绑定：代码仅能在非Windows系统编译，在Windows上功能虚假
   - 部署复杂：部署时需要管理C++动态链接库及其依赖
   - 职责不清：用户服务承担了设备和授权管理的职责，违反单一职责原则

2. **平台强依赖与隐性故障**：在Windows环境下，由于CGO不可用，设备指纹和许可证功能完全失效，形成隐性故障。

## 2. 重构目标

1. **解耦核心业务**：将设备指纹和许可证功能从用户服务中剥离，创建独立的设备认证服务 (`device-auth-service`)。
2. **提升可维护性**：通过服务拆分，使用户服务专注于用户管理核心业务，降低代码复杂度。
3. **改善部署体验**：消除对CGO的强依赖，使用户服务能在任何平台正常编译和运行。
4. **保证功能完整性**：通过内部API调用新的设备认证服务，确保设备指纹和许可证功能在所有平台都能正常工作。

## 3. 重构方案

### 3.1. 创建独立的设备认证服务

我们将创建一个新的微服务 `device-auth-service`，专门负责设备指纹采集、验证和许可证管理功能。

#### 3.1.1. 技术架构

- **语言**：Go + C++ (CGO)
- **通信**：RESTful API
- **部署**：Docker容器化
- **数据库**：PostgreSQL (存储设备和许可证信息)

#### 3.1.2. 目录结构

```
device-auth-service/
├── internal/
│   ├── config/      # 配置管理
│   ├── handlers/    # HTTP处理器
│   ├── middleware/  # 中间件
│   ├── models/      # 数据模型
│   ├── services/    # 业务逻辑
│   └── cpp/         # C++集成代码
├── api/             # API定义
├── config/          # 配置文件
├── docs/            # 文档
├── scripts/         # 脚本
├── Dockerfile       # Docker配置
├── go.mod           # Go模块定义
├── main.go          # 主程序入口
└── Makefile         # 构建脚本
```

#### 3.1.3. 核心功能

1. **设备指纹采集**：通过CGO调用C++库采集设备硬件信息
2. **设备指纹验证**：验证设备指纹的合法性
3. **许可证生成**：为设备生成许可证
4. **许可证验证**：验证许可证的有效性

### 3.2. 修改用户服务

#### 3.2.1. 移除CGO依赖

1. 删除 `user-service/internal/cpp` 目录下的所有C++相关代码
2. 移除 `go.mod` 中与CGO相关的构建标签
3. 修改构建脚本，不再需要CGO环境

#### 3.2.2. 通过内部API调用设备认证服务

1. 在用户服务中添加设备认证服务的客户端
2. 修改需要设备指纹或许可证验证的业务逻辑，通过HTTP请求调用设备认证服务
3. 处理网络异常和超时情况，保证服务的健壮性

#### 3.2.3. 环境适配

1. 添加配置项，支持设备认证服务的地址配置
2. 在Docker Compose中添加设备认证服务的依赖

## 4. 实施步骤

3. 实现RESTful API接口
4. 实现数据库模型和存储逻辑

### 4.2. 第二阶段：重构用户服务

1. 移除用户服务中的C++相关代码
2. 实现设备认证服务的客户端
3. 修改业务逻辑，通过API调用设备认证服务
4. 更新配置文件和构建脚本

### 4.3. 第三阶段：测试和部署

1. 编写单元测试和集成测试
2. 在不同平台测试服务功能
3. 更新部署文档和操作手册

## 5. 风险评估与应对

### 5.1. 技术风险

1. **网络延迟**：通过内部API调用可能引入网络延迟
   - **应对措施**：优化API设计，减少不必要的调用；使用连接池和缓存机制

2. **服务依赖**：用户服务依赖设备认证服务
   - **应对措施**：实现降级策略，在设备认证服务不可用时提供基本功能

### 5.2. 运维风险

1. **部署复杂度增加**：需要同时部署两个服务
   - **应对措施**：提供完整的Docker Compose配置，简化部署流程

2. **监控和日志**：需要监控两个服务的运行状态
   - **应对措施**：实现统一的日志收集和监控方案

## 6. 预期效果

1. **架构清晰**：用户服务和设备认证服务职责分明，符合单一职责原则
2. **平台无关**：用户服务不再依赖CGO，可在任何平台正常编译和运行
3. **易于维护**：代码结构清晰，便于后续功能扩展和维护
4. **功能完整**：设备指纹和许可证功能在所有平台都能正常工作
