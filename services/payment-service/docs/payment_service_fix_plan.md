# 支付服务修复方案

## 1. 问题概述

根据 `docs/archive/audit-reports/final_comprehensive_audit_report_20250905_142256.md` 的综合审计结果，支付服务存在严重问题：
1. 所有支付渠道（微信、支付宝、银联等）均由 `payment_adapter_stub.go` 提供存根(Stub)实现。
2. 这些存根实现不与任何真实支付网关交互，仅返回写死的成功结果。
3. 核心商业闭环完全缺失，无法处理任何真实支付。

## 2. 修复目标

1. 移除 `payment_adapter_stub.go` 中的存根实现。
2. 集成真实的微信、支付宝、银联支付SDK。
3. 实现完整的支付、回调、查询和退款逻辑。

## 3. 详细修复方案

### 3.1. 移除存根实现

- 删除文件 `services/payment-service/internal/adapters/payment_adapter_stub.go`
- 修改 `services/payment-service/internal/adapters/factory.go`，移除对存根适配器的引用，替换为真实的适配器实现。

### 3.2. 集成真实支付SDK

#### 3.2.1. 微信支付

- 已有 `wechat_pay.go` 文件，但需要完善其实现。
- 需要实现完整的退款和回调处理逻辑。
- 确保配置正确加载。

#### 3.2.2. 支付宝

- 已有 `alipay.go` 文件，但需要完善其实现。
- 确保配置正确加载。

#### 3.2.3. 银联支付

- 已有 `unionpay_adapter.go` 文件，但需要完善其实现。
- 确保配置正确加载。

### 3.3. 实现完整逻辑

#### 3.3.1. 支付流程

1. 用户发起支付请求。
2. 系统根据支付渠道调用对应的真实支付SDK。
3. SDK返回支付凭证（如二维码、支付表单等）。
4. 用户完成支付。

#### 3.3.2. 回调处理

1. 支付平台异步通知支付结果。
2. 系统验证回调签名。
3. 更新订单状态。
4. 发送通知给用户。

#### 3.3.3. 查询支付状态

1. 用户或系统主动查询订单支付状态。
2. 调用支付平台API查询。
3. 返回最新状态。

#### 3.3.4. 退款流程

1. 用户发起退款请求。
2. 系统调用支付平台退款API。
3. 更新退款状态。
4. 发送通知给用户。

## 4. 实施步骤

1. 备份现有代码。
2. 删除 `payment_adapter_stub.go` 文件。
3. 修改 `factory.go`，集成真实支付适配器。
4. 完善各支付渠道的实现逻辑。
5. 配置环境变量或配置文件，确保支付参数正确。
6. 编写单元测试和集成测试。
7. 部署到测试环境进行验证。
8. 部署到生产环境。

## 5. 风险与应对

- **风险**: 集成真实支付SDK可能引入新的依赖和兼容性问题。
  - **应对**: 仔细阅读SDK文档，确保版本兼容性。
- **风险**: 支付回调处理不当可能导致状态不一致。
  - **应对**: 严格验证回调签名，确保幂等性处理。
- **风险**: 退款逻辑复杂，可能涉及部分退款、多次退款等情况。
  - **应对**: 仔细设计退款数据结构和处理逻辑，确保准确性。
